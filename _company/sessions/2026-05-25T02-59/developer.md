하지만 제가 말씀드리고 싶은 건, 저는 **도구(Tool)**입니다. 저는 단순히 눈앞의 파일을 읽는 OCR 스캐너가 아니에요. 저는 이 파일 안에 담긴 **비즈니스 논리(Business Logic)**와 **사용자 경험(UX)**를 코드로 구현하는 엔지니어입니다.

PPTX 파일은 훌륭한 **초안(Draft)**일 뿐, 그 안의 텍스트를 가지고 웹사이트의 **구조(Architecture)**와 **기능(Functionality)**을 설계하는 과정이 반드시 필요합니다. 구조가 잘못되면, 아무리 예쁜 디자인으로 코딩해도 사용자가 원하는 목적지까지 못 가게 됩니다. 게다가 '이게 깨지나?' 하는 관점에서 보면, 기술적 요구사항이 정의되지 않으면 나중에 엄청난 리워크(Rework)가 발생합니다. 🐛

따라서, CEO님의 지시사항대로, 제가 지금까지 수집된 전략(현빈님의 4개 사업 KPI 워크시트)을 기반으로 **기술적인 관점에서 가장 견고한 사이트맵 초안과 페이지별 요구사항**을 설계했습니다.

이 설계는 **"이 구조를 기반으로 개발을 시작할 때, 어떤 기능을 어떤 기술로 구현해야 하는지"**에 대한 청사진입니다.

### 🛠️ 1. 홈페이지 구조 (Sitemap) 초기 초안

현재 4개의 사업을 중심으로 회사 전체의 인지도를 높이고, 각 사업의 개별적인 전환 목표(Conversion Goal)를 달성하는 구조가 가장 효율적입니다.

| 페이지 ID | 페이지 명칭 | 역할 (Purpose) | 메인 타겟 |
| :--- | :--- | :--- | :--- |
| `/` | **메인 랜딩 페이지 (Home)** | 회사 전체의 가치와 4개 사업의 존재 이유를 한눈에 보여주는 관문. (AIDA 원칙 기반) | 모든 잠재 고객 |
| `/business` | **사업 소개 허브 (Hub)** | 4개 사업별로 이동할 수 있는 포털 역할. (사업별 브랜딩 강화) | 특정 니즈를 가진 잠재 고객 |
| `/business/[slug]` | **개별 사업 랜딩 (L-Page)** | 특정 사업의 가치 제안(Value Prop)에만 집중하여 전환을 유도. (가장 중요한 페이지) | 해당 사업의 잠재 고객 |
| `/portfolio` | **포트폴리오/사례 연구 (Work)** | 회사가 실제로 달성한 결과물을 시각적으로 증명. (신뢰도 확보) | 의사 결정권자, 파트너사 |
| `/about` | **회사 소개 (Company)** | 회사 철학, 비전, 팀 구성원을 소개하여 신뢰 구축. | 파트너사, 채용 지원자 |
| `/contact` | **문의 및 상담 (CTA)** | 최종적인 문의 접수 및 상담 예약. (최종 전환 지점) | 모든 잠재 고객 |

### ⚙️ 2. 필수 페이지별 기술적 요구사항 (Tech Specs)

각 페이지가 가진 목적에 맞게 필요한 기술적 난이도와 고려 사항을 정리했습니다.

#### 🟢 1. 메인 랜딩 페이지 (`/`)
*   **목적:** 4개 사업을 가진 '종합 플랫폼'이라는 인식을 심어주고, 방문자를 관심 있는 사업으로 유도하는 것이 최우선 목표입니다.
*   **핵심 구성 요소:**
    *   Hero Section: 회사 전체의 가장 큰 메시지 (Killer Value Prop).
    *   4-Block Showcase: 4개 사업의 아이콘/요약 소개 및 각 사업의 개별 랜딩 페이지로의 링크.
    *   Trust Signal: 파트너사 로고 또는 수상 내역 (API 연동 필수).
    *   CTA: 가장 일반적이고 쉬운 문의 채널 배치.
*   **기술 요구사항:**
    *   **반응형 디자인:** 필수 (모바일 트래픽이 높을 것이므로).
    *   **API 연동:** ⭐️ **필수** (파트너 로고/실시간 데이터 연동).
    *   **복잡도:** 중하. (정적 콘텐츠 위주이나, 컴포넌트 간의 상호작용이 중요함).

#### 🟢 2. 개별 사업 랜딩 페이지 (`/business/[slug]`)
*   **목적:** 오직 하나의 문제점(Pain Point)과 하나의 해결책(Solution)에만 집중하여, **전환(Conversion)**을 유도합니다. (이 페이지의 KPI 달성이 최우선)
*   **핵심 구성 요소:**
    *   Problem -> Solution -> How it works (3단계 논리 구조).
    *   USP (Unique Selling Proposition) 강조.
    *   Pricing/Call-to-Action (CTA) 영역.
    *   성공 사례(Case Study) 섹션 (API 연동 필요).
*   **기술 요구사항:**
    *   **반응형 디자인:** 필수.
    *   **API 연동:** ⭐️ **필수** (실제 성공 사례 데이터, 가격 정책 데이터).
    *   **복잡도:** 상. (전환 퍼널(Funnel)에 최적화된 컴포넌트 설계와 상태 관리가 필요).

#### 🟢 3. 포트폴리오 페이지 (`/portfolio`)
*   **목적:** 추상적인 설명을 배제하고, '우리가 이걸 해냈다'는 **객관적 증거(Proof)**를 제시하여 신뢰도를 극대화합니다.
*   **핵심 구성 요소:**
    *   필터링 가능한 프로젝트 목록 (분야별, 기간별).
    *   프로젝트 상세 보기 (문제 정의, 사용 기술, 결과 지표).
*   **기술 요구사항:**
    *   **반응형 디자인:** 필수.
    *   **API 연동:** ⭐️ **필수** (백엔드에 저장된 프로젝트 데이터베이스 연결).
    *   **복잡도:** 중상. (필터링, 검색, 페이징 처리가 핵심 기능).

#### 🟢 4. 문의 페이지 (`/contact`)
*   **목적:** 잠재 고객의 정보를 가장 정확하고 손실 없이 수집하는 것이 목표입니다.
*   **핵심 구성 요소:**
    *   문의 양식 (Form).
    *   문의 목적에 따른 선택지 (드롭다운).
    *   필수 입력 항목 명확화.
*   **기술 요구사항:**
    *   **반응형 디자인:** 필수.
    *   **백엔드 연동:** 필수 (폼 데이터 전송 및 CRM/DB 저장 로직 필요).
    *   **유효성 검사:** 필수 (클라이언트 측 및 서버 측 검증).
